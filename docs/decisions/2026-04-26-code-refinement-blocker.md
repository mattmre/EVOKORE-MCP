## Background

The overnight panel-of-panels run flagged a "verified blocker" for the
upcoming Wave 3 skill-composition-graph sprint:

> `code-refinement` skill does not exist as an executable skill — only
> as a panel definition under
> `SKILLS/ORCHESTRATION FRAMEWORK/panel-of-experts/panels/code-refinement.md`.
> The `pr-manager → security-review → code-refinement` chain referenced
> in prose is aspirational, not wired.

The implementer was given two options:
- **(a)** Build a minimal executable `code-refinement` skill so the chain
  is real.
- **(b)** Remove the dangling references so prose matches reality.

## Decision — option (b): remove the dangling reference

Selected option (b) for the following reasons:

1. **The panel definition is already a skill.** The
   `panel-of-experts/panels/code-refinement.md` file has proper
   frontmatter (`name: panel-code-refinement`) and is loaded by
   `SkillManager.loadSkills()` (verified by the existing snapshot test
   for default-profile search). There is no missing artifact —
   only a name-shape mismatch in the orchestrator's prose. Building a
   *separate* executable `code-refinement` skill on top would create a
   duplicate concern and a naming collision with the panel.
2. **The aspirational chain has no wiring.** Neither `pr-manager` nor
   `security-review` invokes a code-refinement step. The chain only
   exists in three planning documents authored in one overnight session.
   Building a skill to satisfy the prose would invert the dependency
   direction (skill exists to satisfy doc, instead of doc reflecting
   skill).
3. **Minimum maintenance burden.** Option (b) deletes drift, option (a)
   would add a new skill file plus an injection point in `pr-manager`,
   plus tests for both. The plan explicitly characterises this slice as
   "audit + repair," not feature build.
4. **Wave 3 skill-composition-graph unblocked.** The graph derivation
   script in Sprint 2.x will scan `SKILLS/**/SKILL.md` for
   `invoke .*-skill|run .*panel` patterns. After this PR, the
   `panel-of-experts` SKILL.md continues to surface `panel-code-refinement`
   under the real, loadable skill name; the planning prose no longer
   describes a chain that the graph would fail to resolve.

## Migration note for operators

If an operator was relying on the prose description of a
`pr-manager → security-review → code-refinement` chain, the working
equivalent today is:

1. Run the `pr-manager` skill (or `orch-review`).
2. When the risk-routing table flags security-relevant changes, invoke
   the `security-review` skill.
3. For multi-perspective code-quality review, run the
   **panel-of-experts** code-refinement panel via:
   - `orch-panel code <artifact>`, or
   - the natural-language invocation documented in
     `panel-of-experts/SKILL.md` under "Direct Invocation".

The panel skill is registered under the name `panel-code-refinement`
(not bare `code-refinement`) — that is the canonical handle for any
future automation, including the Sprint 2.x skill-composition-graph
derivation script.
