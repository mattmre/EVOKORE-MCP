# EVOKORE-MCP Developer Context & Learnings

> See SOUL.md for behavioral identity. See RULES.md for declarative security policies.

## System Architecture
- EVOKORE-MCP v3.0 acts as a Multi-Server MCP Aggregator proxying to child servers (e.g., github, fs, supabase) defined in `mcp.config.json` with a JSON-RPC router.
- Tool-prefixing is active to prevent namespace collisions.

## Key Workflows & Learnings
- **Human-in-the-Loop (HITL):** Since JSON-RPC via stdio is stateless and the server cannot natively prompt the user in the AI client's terminal, EVOKORE uses an `_evokore_approval_token`. Restricted tools return an error directing the AI to ask the user and then retry with the token injected into the tool arguments.
- **Submodules:** The repository uses git submodules for SKILLS (`ANTHROPIC COOKBOOK`, `claude-skills-mcp`, `OFFICIAL MCP SERVERS`). 
- **Doc Generation Scripts:** Scripts like `scripts/generate_docs.js` add frontmatter and modify files within these submodules. Always commit changes inside the submodules first before updating the parent repo pointer to avoid `-dirty` submodule states.
- **Hook Scripts:** Canonical Claude Code hook entrypoints now live under `scripts/hooks/*.js`, and `.claude/settings.json` points at those paths. Legacy top-level `scripts/*.js` wrappers still delegate to the same runtime. `scripts/status.js` remains available for live CLI status integrations.
- **Path Resolution:** Remember that compiled TypeScript files run out of the `dist/` directory! Relative paths from managers resolving to root files (e.g., `mcp.config.json`, `.env`, `SKILLS/`) must resolve to `../`, not `../../`.
- **Regex Parsing:** When using JavaScript's native RegExp for matching markdown frontmatter, avoid using string `new RegExp` constructor with multiple escapes (`\\\\r`), which evaluates to literal slashes. Prefer RegExp literals like `/^---\r?\n([\s\S]*?)\r?\n---\r?\n([\s\S]*)$/`.
- **Documentation Overhauls:** When orchestrating documentation updates using `docs_architect` or multiple workflows concurrently, ensure that legacy or theoretical endpoint references (e.g., `/docs/architecture.md`) are explicitly mapped to their physical canonical equivalents (e.g., `docs/V2_MULTI_AGENT_WORKFLOWS.md`) in `docs/README.md` to prevent context rot.
- **Docs Portal Cross-Links:** If a refreshed canonical docs page links to dated research or session-log artifacts, publish those linked artifacts in the same docs PR or keep the links generic until the artifacts exist. `test-docs-canonical-links.js` will fail fast on missing dated targets.
- **Windows Executable Resolution:** `npx` requires `.cmd` suffix on Windows for `child_process.spawn`; `uvx`/`uv` install as native `.exe` and resolve via PATH through `cross-spawn` without needing `.cmd`. Don't blindly add `.cmd` to all commands.
- **Env Variable Interpolation:** `mcp.config.json` supports `${VAR}` syntax in `env` blocks. The ProxyManager resolves these from `process.env` (loaded via dotenv from `.env`) before passing to child server processes.
- **Voice Integration:** ElevenLabs MCP is proxied as a child server (24 tools with `elevenlabs_*` prefix). VoiceMode is user-scoped in Claude Code (needs direct mic/speaker access). See `docs/VOICE_CLI_RESEARCH.md` for full architecture research and `docs/USAGE.md` Section 4 for setup.
- **VoiceMode Windows Bug:** The `voice-mode-install` script crashes on Windows due to a Unicode encoding error (cp1252 codec). Skip the installer and use `claude mcp add` directly.
- **Voice Sidecar:** `VoiceSidecar.ts` is standalone (never imported by `index.ts`). It runs as a separate process on `ws://localhost:8888`. Audio playback uses temp files + platform players (no native deps). `voices.json` supports hot-reload (re-read per connection). Keep `voices.json` documentation and examples limited to keys actually consumed at runtime; do not imply TypeScript-only provider fields are valid persona config keys.
- **Voice Hook Persona Routing:** `scripts/voice-hook.js` now targets loopback by default and forwards persona through `VOICE_SIDECAR_PERSONA` first, then payload persona metadata. Use that explicit hook-side control when you want non-default voice personas in the normal Claude stop-hook path.
- **Cross-CLI Sync:** `scripts/sync-configs.js` merges the `evokore-mcp` entry into each CLI's config, preserves existing entries by default, and resolves the canonical git common root so disposable worktrees do not register temporary `dist/index.js` paths. Use `--dry-run` to preview changes. Use `EVOKORE_SYNC_PROJECT_ROOT` only when intentionally overriding the detected repo root.
- **Environment Validation:** When validating `.env` configurations, use the `test-dotenv-quiet-validation.js` script to ensure quiet parsing is enforced.
- **Git Worktree Cleanup:** Abandoned agent worktrees can accumulate (e.g., 9 found previously). Run `git worktree list` and `git worktree remove <path>` aggressively when doing branch cleanups. If repo state looks impossible after a crash, inspect `.git/config` for null-byte corruption before doing broader branch or worktree surgery.
- **Merged Branch Pruning From Dirty Root Branches:** If the active handoff branch is dirty and not based on `main`, `git branch -d` can refuse to delete branches that are already merged into `origin/main`. Verify with `git merge-base --is-ancestor <branch> origin/main`, then use `git branch -D <branch>` only for confirmed already-landed history.
- **Repo State Audit:** `scripts/repo-state-audit.js` provides a human-readable and `--json` audit of current-branch divergence, live worktrees, stale local branches with gone upstreams, merged remote branch candidates without open PRs, open PR heads, and control-plane drift. After PR `#105` lands, use `npm run repo:audit` before cleanup or new multi-slice work.
- **Post-Merge Cleanup Sequence:** After merging a PR that changes shared repo surfaces, refresh the next PR branch onto `origin/main`, rerun local validation, then wait for fresh GitHub checks before merging. Apply the same sequential rule to non-overlapping validation/docs PR waves too; batch-merging "independent" PRs still increases CI drift and handoff mistakes. This was required for PR `#105` after PR `#104` landed and again during the 2026-03-26 validation merge wave.
- **Dirty Root Branch Retirement:** If the root handoff worktree is parked on a stale feature branch, switch it onto a fresh handoff branch based on `origin/main` before deleting the stale local branch. This let the repo keep its dirty control plane without leaving the root worktree on obsolete history.
- **Control-Plane Preservation PRs:** If the remaining root drift is only trackers, research notes, or session logs, preserve it on a dedicated `chore/control-plane-*` branch/PR before continuing cleanup. That keeps the root worktree clean without discarding handoff state.
- **Stale Feature Replay:** If a branch predates major runtime/config changes, port the valid feature commits onto a fresh `main`-based worktree instead of reviving the stale branch in place. This prevented `feature/damage-control-expansion` from reintroducing legacy `.claude/settings.json` wiring and package regressions.
- **Git Porcelain Parsing:** When parsing `git status --short`, do not `trim()` the full command output. That strips the first line's leading status whitespace and corrupts XY/path parsing. Remove trailing newlines only.
- **Remote Branch Cleanup:** Run `git fetch --prune origin` before deleting remote branches after a merge wave. GitHub may have already auto-deleted some PR heads, and pruning avoids false cleanup failures.
- **GitHub CLI JSON Fields On Windows:** In this shell environment, pass the `--json` field list to `gh pr view` / `gh pr list` as one quoted argument (for example, `--json 'number,title,state'`) or PowerShell will split the fields into separate arguments.
- **PR Review Coverage Audits:** When auditing historical GitHub review coverage, distinguish `comments` from submitted `reviews`. Automated PR summary or quota-warning comments do not mean a PR has an actual review record.
- **Session-Wrap Review Drift:** When reviewing docs/control-plane PRs, explicitly check for stale placeholders, incorrect status labels, and overly broad wording even if CI is green. Handoff artifacts can be semantically wrong while remaining syntactically valid.

- **Orchestration Conflicts:** When orchestrating multiple agents in parallel, ensure they do not commit shared ephemeral tracking logs (like `session-logs/*.md` or `next-session.md`) to their feature branches. Committing shared trackers on feature branches causes unresolvable git merge conflicts when sequentially squashing PRs. Trackers should be updated directly on `main` after all PRs are merged.
- **Orchestrator-Imagined Skill Chains:** Multi-panel overnight runs can fabricate skill chains in plan/handoff prose (e.g., "`pr-manager → security-review → code-refinement`") even when no skill body actually invokes the next link. Before treating such a chain as a "blocker," grep the implicated SKILL.md bodies — if neither side names the next link, the chain is prose drift, not a real wiring gap. Fix the prose, not the skills. See `docs/decisions/2026-04-26-code-refinement-blocker.md` for the canonical example.
- **Damage Control Regex:** The fork bomb detection regex in `damage-control-rules.yaml` must use `:\(\)\s*\{` not `:(\\){0}){2,}`. The latter evaluates to matching any colon character due to `{0}` making `)` match nothing. Always test regex patterns with `new RegExp(pattern).test(":")` to verify they don't match trivially.
- **Commit Message Approach:** Use `.commit-msg.txt` file with `git commit -F .commit-msg.txt` instead of heredocs or inline `-m` flags, because the damage-control hook can misfire on complex command strings. Delete the file after commit.
- **Agent33 Orchestration Framework:** Imported under `SKILLS/ORCHESTRATION FRAMEWORK/` with sub-skills: handoff-protocol (15 docs), policy-pack-v1 (14 docs), commands (11 orch-* skills), workflow-templates (8 DAG JSON), agent-archetypes (6 specs), tool-governance (8 specs), aep-framework (6 docs), improvement-cycles, coding-reference, schemas (3 JSON). Migration complete (PRs #71-#80 merged).
- **Regex Backtracking in Tests:** Avoid `(?:[^\n]*\r?\n)*?` in test regexes matching YAML/markdown — causes catastrophic backtracking on multi-section files. Use direct section matching instead (e.g., `/\btest:\s*\r?\n\s*name:/`).
- **Git Lock Files:** When damage-control blocks `rm -f` on `.git/index.lock`, use `unlink .git/index.lock` instead. Parallel agent worktrees can also leave stale lock files — always check after worktree-based agent runs.
- **PR Template:** Uses `.github/PULL_REQUEST_TEMPLATE.md` (uppercase) with section-based format (`## Description`, `## Type of Change`, `## Testing`, `## Evidence`). CI validation in `validate-pr-metadata.js` checks for these sections. If the PR body is fixed after opening the PR, GitHub may keep validating the stale event payload until a new synchronize event occurs, so push a fresh sync commit if the metadata job stays red. When the PR body mentions `.env`, prefer `gh pr create --body-file` or `gh pr edit --body-file` to avoid damage-control false positives from inline shell strings. In this PowerShell environment, `--body-file -` was not reliable for PR creation/editing during PR `#211`; use a real temp file when the CLI appears to drop the body.
- **Release Preflight:** Before tagging a release, verify the target `package.json` version and confirm `NPM_TOKEN` is present if npm publication is expected. GitHub release publication can succeed while npm publish is skipped, so treat those as separate gates.
- **Release Closure State:** A clean `npm run release:preflight` on current `origin/main` is now healthy apart from two release-state facts: the `v3.1.0` git tag / GitHub release already exist, and npm publication is still absent because `NPM_TOKEN` is missing or unconfirmed. `npm view evokore-mcp version` returning `404` is the quickest external confirmation that npm publish has not happened.
- **Skill Indexing:** `SkillManager.loadSkills()` is recursive now. The merged index currently covers roughly 336 skills/files, not the old ~47 two-level subset. Local startup indexing is sub-second on the current corpus, and performance monitoring is layered on top of the recursive shape.
- **Semantic Resolution:** `resolve_workflow` now uses aliases, semantic hints, and reranking to prefer actionable top-level skills over deep reference leaves, and it emits `Why matched:` explanations for returned skills.
- **Session Continuity Manifest:** Runtime continuity is now anchored in `~/.evokore/sessions/{sessionId}.json`. `purpose-gate`, `session-replay`, `evidence-capture`, and `tilldone` all update this shared manifest while replay/evidence/task files remain their append-only sibling artifacts. `T19` and `T21` should build on this manifest instead of inferring state separately from each artifact file.
- **Claude Memory Sync:** `npm run memory:sync` now manages the live Claude project memory directory for EVOKORE at `C:\Users\mattm\.claude\projects\D--GITHUB-EVOKORE-MCP\memory`. The generated set is `MEMORY.md`, `project-state.md`, `patterns.md`, and `workflow.md`, and it uses repo-aware session matching via `workspaceRoot` + `canonicalRepoRoot`.
- **Status Runtime:** `scripts/status.js` is now backed by `scripts/status-runtime.js` and should remain continuity-first: branch/worktree pressure, session purpose, task pressure, continuity health, and context usage come from the canonical session manifest plus git state, with Claude memory only as fallback. `purpose-gate` reuses the same shared summary path, so any future status change must update both surfaces together.
- **Log Rotation:** `scripts/log-rotation.js` provides shared `rotateIfNeeded(filePath, opts)` and `pruneOldSessions(sessionsDir, opts)`. All 5 hook scripts that write logs use this module. Default: 5MB max size, 3 rotations; sessions: 30 day max age, 100 file max.
- **Tracker Validation Sync:** When archiving or restructuring tracker docs (`ORCHESTRATION_TRACKER.md`, `PRIORITY_STATUS_MATRIX.md`), the corresponding validation scripts (`scripts/validate-tracker-consistency.js` `REQUIRED_TRACKER_SECTIONS`, `test-ops-docs-validation.js`) must be updated to match the new section names — the crashed session missed this, causing test failures.
- **Docs Archive:** Historical orchestration execution logs and evidence refresh narratives are archived in `docs/archive/`. Active trackers contain only templates, checklists, and current-state summaries.
- **Registry Config Path Alignment:** `SkillManager` must honor `EVOKORE_MCP_CONFIG_PATH` just like `ProxyManager`. Temp-config tests and runtime overrides should never depend on a hard-coded repo-local `mcp.config.json`.
- **Registry Listing Consistency:** Do not maintain separate fetch/parsing logic for `list_registry` and `listRegistrySkills()`. Both surfaces should share the `RegistryManager` path so `{ entries: [...] }`, `{ skills: [...] }`, cache behavior, and URL normalization stay aligned.
- **Registry Docs Schema:** Active docs must show `skillRegistries` as objects with `name`, `baseUrl`, and `index`. Bare string-array examples drift from runtime and create false operator validation failures.
- **Raw Import Cleanup:** If a copied skill pack or sidecar drop has already been normalized into a clean feature PR, remove the duplicate raw root copy during repo cleanup instead of trying to cherry-pick both versions forward.
- **FileSessionStore Restart Scope:** Historical note: PR `#175` only proved persistence across a fresh `FileSessionStore` + `SessionIsolation` boundary and did not by itself implement runtime HTTP reattachment. That later gap was closed in M1.1 / PR `#194`, which wires `SessionIsolation.loadSession()` into `HttpServer`. Do not cite the old PR `#175` evidence alone as proof of runtime continuity.
- **Audit Redaction Path:** If audit metadata redaction is a requirement, wire it through the shared `AuditLog.write()` path rather than relying on helper-level tests or per-call-site wrappers. PR `#207` closed post-M2 finding F1 this way.
- **Webhook Event System:** `src/WebhookManager.ts` dispatches HMAC-SHA256 signed events (tool_call, tool_error, session_start, session_end, approval_requested, approval_granted) to HTTP endpoints configured in `mcp.config.json` under the `webhooks` key. Fire-and-forget with 3 retries + exponential backoff. Opt-in via `EVOKORE_WEBHOOKS_ENABLED=true`. Tool arguments are redacted for sensitive keys before emission. `verifySignature()` uses `crypto.timingSafeEqual` for timing-safe verification. `getWebhooks()` returns sanitized configs (no raw secrets).
- **Plugin System:** `src/PluginManager.ts` loads plugins from `plugins/` directory (configurable via `EVOKORE_PLUGINS_DIR`). Plugins export tools that get prefixed and registered. `reload_plugins` tool hot-reloads at runtime. Plugin tools are dispatched through the standard CallToolRequest handler.
- **OAuth Bearer Token Auth:** `src/auth/OAuthProvider.ts` validates Bearer tokens on HTTP transport requests. Supports JWKS-based key rotation. Configure via `EVOKORE_OAUTH_ISSUER`, `EVOKORE_OAUTH_AUDIENCE`, `EVOKORE_OAUTH_JWKS_URI`.
- **StreamableHTTP Server Transport:** `src/HttpServer.ts` serves MCP over HTTP with SSE streaming. Listens on `EVOKORE_HTTP_PORT` (default 3100) and `EVOKORE_HTTP_HOST` (default 127.0.0.1). Supports both stdio and HTTP modes via `--http` flag.
- **Multi-Tenant Session Isolation:** `src/SessionIsolation.ts` provides per-session state isolation for HTTP transport. Each connection gets independent tool activation state and session context. Configurable TTL via `EVOKORE_SESSION_TTL_MS`.
- **PR Review Security Hardening:** When reviewing webhook/event systems, always check: (a) timing-safe HMAC comparison, (b) argument/payload redaction for secrets, (c) URL scheme validation (HTTP/HTTPS only), (d) double-resolve guards in Promise-based delivery, (e) semantic alignment between docs and implementation for event timing (invocation vs completion).
- **GitHub Actions Quota:** During heavy sprint sessions (10+ PRs in one session), GitHub Actions minutes can be exhausted. Check quota before expecting CI to run. Local `npx vitest run` is the fallback validation.
- **Squash Merge Branch Cleanup:** When PRs are squash-merged, local branches are NOT git ancestors of main. Use `git branch -D` (force) after confirming the PR is merged on GitHub, not `git branch -d` which checks ancestry.
- **Self-Approval Limitation:** GitHub blocks self-approval on your own PRs. When doing a PR-manager self-review pass, record the review result as a normal PR comment instead of an approval review.
- **Node CLI Syntax Checks In Tests:** For Node/CommonJS CLI scripts, prefer `spawnSync(process.execPath, ['--check', scriptPath])` over `new Function(source)` in validation tests. `new Function` parses as a function body and can reject valid Node CLI syntax; PR `#208` fixed this for `test-worktree-cleanup-validation.js`.
- **UTF-8 BOM in .claude/settings.json:** If `.claude/settings.json` is ever edited by Windows tools (PowerShell `Out-File`, Notepad), a UTF-8 BOM (`EF BB BF`) may be prepended. Linux CI fails immediately with `SyntaxError: Unexpected token '﻿'` in `hook-e2e-validation.js` and `test-repo-audit-hook-validation.js`. Fix by rewriting the file with the Write tool (which writes clean UTF-8). Verify with `xxd .claude/settings.json | head -1` — first bytes must be `7b0a` (no `efbbbf` prefix).
- **Env Sync Validation:** `test-env-sync-validation.js` (shard 3) scans `src/` for `process.env.EVOKORE_*` references and requires each to be documented in `.env.example`. When adding a new env var to any `src/*.ts` file, always add the matching commented-out example line to `.env.example` in the same commit/PR, or CI shard 3 will fail.
- **Damage Control .env.example False Positive:** Damage-control blocks Bash commands containing `.env` as a path substring (matches `.env.example`). When needing to `git add .env.example`, use `git add -A` or stage other files individually without typing `.env` in the command string.
- **PR Rebase Order for Shared Fixes:** If PR A fixes tests that PR B also needs (e.g., `execFileSync` → `execFileAsync` rename, `.env.example` additions), merge PR A first, then rebase PR B onto the updated main. Do not try to fix the same issue in both PRs independently — the duplication creates conflicts.
- **VoiceSidecar Playback Queue:** Multiple Claude Code sessions firing the Stop hook simultaneously cause overlapping audio. Fix: serialize playback via a `playbackQueue` + `drainPlaybackQueue()` pattern in `VoiceSidecar.ts`. The `finalize()` method enqueues instead of playing inline.
- **Progressive Disclosure for Skill Authoring:** When writing EVOKORE SKILLS, use progressive disclosure: surface the trigger condition and 2-3 sentence summary first, then detailed sections. A reader should be able to decide in 5 seconds whether the skill applies to their task. Avoid leading with implementation details.
- **Trigger-Explicit Skill Descriptions:** The `description` frontmatter field must state *what action* triggers the skill (e.g., "Use when automating browser interactions..."), not a vague noun phrase (e.g., "Browser automation tool"). This enables accurate semantic resolution via `resolve_workflow`.

## Claude Code Hooks

Seven hooks are wired in `.claude/settings.json` through canonical `scripts/hooks/*.js` entrypoints. All active entrypoints share `scripts/hooks/fail-safe-loader.js` so load failures degrade safely instead of crashing Claude Code. State lives in `~/.evokore/sessions/` and logs in `~/.evokore/logs/`.

- **Damage Control** (`scripts/hooks/damage-control.js`, PreToolUse): Security auditor that blocks dangerous commands and sensitive path access. Rules defined in `damage-control-rules.yaml`. Fail-open if rules file is missing.
- **Purpose Gate** (`scripts/hooks/purpose-gate.js`, UserPromptSubmit): Asks for session intent on first prompt, then injects purpose reminders via `additionalContext` to keep sessions focused.
- **Session Replay** (`scripts/hooks/session-replay.js`, PostToolUse): Logs all tool usage as JSONL to `~/.evokore/sessions/{session_id}-replay.jsonl`. View with `npm run replay` or `node scripts/session-replay-view.js <session_id>`.
- **TillDone** (`scripts/hooks/tilldone.js`, Stop): Blocks session stop if incomplete tasks remain. Also a standalone CLI:
  ```
  node scripts/tilldone.js --add "task text" --session <ID>
  node scripts/tilldone.js --done 1 --session <ID>
  node scripts/tilldone.js --toggle 1 --session <ID>
  node scripts/tilldone.js --list --session <ID>
  node scripts/tilldone.js --clear --session <ID>
  ```
- **Evidence Capture** (`scripts/hooks/evidence-capture.js`, PostToolUse): Auto-captures test results, file changes, and git operations as JSONL evidence to `~/.evokore/sessions/{session_id}-evidence.jsonl`. Sequential evidence IDs (E-001, E-002, etc.).
- **Repo Audit** (`scripts/hooks/repo-audit-hook.js`, UserPromptSubmit): Pre-session hook that warns about branch drift, stale worktrees, and control-plane drift. Enabled by default; opt-out via `EVOKORE_REPO_AUDIT_HOOK=false`. Runs once per session (marker file).
- **Voice Stop** (`scripts/hooks/voice-stop.js`, Stop): Fires voice sidecar finalization on session end.

## v3.0 Runtime Additions

- **Default Discovery Mode (v3.1+):** unset `EVOKORE_TOOL_DISCOVERY_MODE` defaults to `dynamic` (lean tools/list — native tools always visible, proxied tools activated on demand via `discover_tools`). Set `EVOKORE_TOOL_DISCOVERY_MODE=legacy` to restore the pre-v3.1 behavior where every proxied tool ships in the initial `tools/list`.
- **Discovery Profiles (Sprint 1.4):** `EVOKORE_DISCOVERY_PROFILE=<name>` selects a named profile from `discovery.profiles` in `mcp.config.json`. Five presets ship by default: `coding` (~1.6K tokens), `research` (~1K), `voice` (~0.8K), `legacy-full` (full pre-v3.1 surface), and `legacy-dynamic` (v3.0 default). `EVOKORE_TOOL_DISCOVERY_MODE=legacy` remains a hard safety pin that overrides any profile selection.
- **Discovery Profile Benchmark:** `node scripts/benchmark-tool-discovery.js --all` (or `--profile <name>`) emits per-profile token counts using the real `js-tiktoken` `cl100k_base` encoding. See [`docs/TOOL_DISCOVERY_PROFILES.md`](docs/TOOL_DISCOVERY_PROFILES.md) for measured budgets, deprecation timeline, and customization guidance.
- **Test Suite:** Tests now use vitest (121 files, ~2053 tests as of v3.1 sprint). Run with `npx vitest run`. Old `node test-*.js` chaining is removed.
- **Tool Annotations:** All 34 native tools have MCP annotations (`readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint`) and `title` fields.
- **Native Tool Count:** 34 built-in tools: 12 in SkillManager (docs_architect, skill_creator, resolve_workflow, search_skills, get_skill_help, discover_tools, proxy_server_status, refresh_skills, fetch_skill, list_registry, execute_skill, describe_tool), 2 in TelemetryManager (get_telemetry, reset_telemetry), 1 in PluginManager (reload_plugins), 2 in NavigationAnchorManager (nav_get_map, nav_read_anchor), 3 in SessionAnalyticsManager (session_context_health, session_analyze_replay, session_work_ratio), 4 in ClaimsManager (claim_acquire, claim_release, claim_list, claim_sweep), 3 in MemoryManager (memory_store, memory_search, memory_list), 4 in FleetManager (fleet_spawn, fleet_claim, fleet_release, fleet_status), 3 in OrchestrationRuntime (orchestration_start, orchestration_stop, orchestration_status). Canonical source: `getTools()` methods in each manager class. Plugin tools are dynamic and not counted here. Last verified: 2026-04-26.
- **Server Instructions:** The MCP Server constructor includes an `instructions` string for client-side display.
- **HTTP Client Transport:** ProxyManager supports `StreamableHTTPClientTransport` for HTTP-based child servers. Configure with `"transport": "http"` and `"url"` in mcp.config.json.
- **MCP Resources:** `resources/list` returns skills as `skill://` URIs plus server-level resources (`evokore://server/status`, `evokore://server/config`, `evokore://skills/categories`).
- **MCP Prompts:** `prompts/list` returns 3 prompts: `resolve-workflow`, `skill-help`, `server-overview`.
- **Skill Hot-Reload:** `refresh_skills` tool rescans SKILLS/ directory. Optional filesystem watcher via `EVOKORE_SKILL_WATCHER=true`.
- **`tools/list` cursor pagination (v3.1+):** the handler supports MCP cursor pagination but defaults to **off** to preserve pre-v3.1 single-call behavior. Pagination activates either when the client sends a `cursor` param (signaling cursor support) or when the operator forces it on with `EVOKORE_TOOL_LIST_PAGINATION=on`. Default page size is 35 (under the Cursor IDE 40-tool cap), tunable via `EVOKORE_TOOL_LIST_PAGE_SIZE` clamped to `[1, 1000]`. Cursors are auto-invalidated on `tools/list_changed`: stale cursors decode to a graceful first-page reset rather than throwing.
- **Schema-deferred `tools/list` + `describe_tool` (Sprint 3.x, opt-in):** `EVOKORE_TOOL_SCHEMA_MODE=deferred` strips per-tool `inputSchema` from `tools/list` (placeholder + `_meta.schema_deferred`), and the always-visible native `describe_tool({tools: string[]}) → {schemas, unknown}` is the bootstrap path. If no client invokes `describe_tool` within `EVOKORE_TOOL_SCHEMA_FALLBACK_MS` (default 60000), the runtime auto-reverts to `full` mode and emits a one-time stderr warning + `tools/list_changed`. Default mode is `full`. Hard-advisory: do not enable for Cline, Claude Code, Cursor, or Claude Desktop — see `docs/research/tools-list-deferred-schema-compat-2026-04-26.md`.
- **Skill composition graph + `nextSteps[]` (Sprint 2.x):** `scripts/derive-skill-composition.js` (run via `npm run skill-graph`) statically scans `SKILLS/**/SKILL.md` bodies for invocation phrases (`invoke X skill`, `run X panel`, etc.) and parses the `## Injection Points` table out of `SKILLS/ORCHESTRATION FRAMEWORK/panel-of-experts/SKILL.md` to emit a `skill-graph.json` build artifact. `execute_skill` lazy-loads that graph (async `fs/promises`, mtime-based reload) and returns `nextSteps: [{ skill, reason }]`; `index.ts` auto-activates any matching tool entries (with kebab-case → snake_case fallback) and emits exactly one `tools/list_changed`. Cycles are detected (DFS) and transitive edges are restricted to a 7-skill allowlist (`release-readiness`, `repo-ingestor`, `docs-architect`, `orch-review`, `orch-plan`, `tool-governance`, `orch-refactor`) capped at depth 5. Override the artifact path with `EVOKORE_SKILL_GRAPH_PATH`.
- **Tool-discovery tiering complete (2026-04-27):** all phases of [`docs/plans/tool-discovery-tiering-2026-04-26.md`](docs/plans/tool-discovery-tiering-2026-04-26.md) shipped on `main` (PRs #288–#299). Token-budget knobs are `EVOKORE_TOOL_DISCOVERY_MODE` (legacy/dynamic), `EVOKORE_DISCOVERY_PROFILE` (named presets), `EVOKORE_TOOL_LIST_PAGINATION` (cursor pagination), and `EVOKORE_TOOL_SCHEMA_MODE` (full/deferred). Defaults preserve pre-v3.1 client compatibility unless an operator opts in.
- **Rate Limiting:** Configurable per-server and per-tool rate limits via `rateLimit` in mcp.config.json. Token bucket algorithm, separate from error-triggered cooldown.
- **Session Dashboard:** `npm run dashboard` launches a zero-dependency web UI on `127.0.0.1:8899` for session replay and evidence viewing.
- **HITL Approval UI:** Dashboard `/approvals` page shows pending approval tokens with deny buttons. File-based state at `~/.evokore/pending-approvals.json`.
- **Skill Versioning:** Optional `version`, `requires`, `conflicts` fields in skill frontmatter. `validateDependencies()` checks satisfaction.
- **Remote Skill Fetch:** `fetch_skill` tool downloads skills from URLs. `list_registry` tool reads configured `skillRegistries` from mcp.config.json.
- **Skill Execution Sandbox:** `execute_skill` tool extracts code blocks from skills and runs them with 30s timeout, 1MB output limit. Supports bash, js, python, ts.
- **Supabase Integration:** Supabase MCP added as proxied child server with tiered permissions (10 allow, 4 require_approval, 3 deny).
- **RBAC Permissions:** Role-based permission model with `admin`, `developer`, `readonly` roles. Activated via `EVOKORE_ROLE` env var. Backwards-compatible when unset (flat permissions still work).
- **Build Hygiene:** Compiled artifacts no longer tracked in `src/`. Only `dist/` has compiled output, and it's gitignored.
- **Async Proxy Boot:** `ProxyManager.loadServers()` runs asynchronously in the background so the MCP handshake completes immediately. The boot emits `"Proxy bootstrap complete"` or `"Background proxy bootstrap failed"` sentinels to stderr. Use `.catch()` on the background promise, never `void` fire-and-forget, so boot errors are visible. Integration tests that call proxied tools must await `waitForProxyBoot(transport)` from `tests/helpers/wait-for-proxy-boot.js` after `client.connect()` before asserting on tool results. Boot timeout is configurable via `EVOKORE_CHILD_SERVER_BOOT_TIMEOUT_MS` (default 30000).
- **Webhook Events:** `WebhookManager` emits events to configured HTTP endpoints with HMAC signatures. 10 event types (tool_call, tool_error, session_start, session_end, session_resumed, approval_requested, approval_granted, plugin_loaded, plugin_unloaded, plugin_load_error), fire-and-forget delivery.
- **Plugin System:** `PluginManager` loads external tool providers from `plugins/` directory with hot-reload support.
- **OAuth Auth:** `OAuthProvider` validates Bearer tokens with JWKS support for HTTP transport.
- **Multi-Tenant Sessions:** `SessionIsolation` provides per-connection state isolation with configurable TTL.
- **AI Navigation Anchors:** `src/NavigationAnchorManager.ts` exposes `nav_get_map` (scan a file for `@AI:NAV` structured comments, return section map at ~100 tokens vs 4K–40K for full read) and `nav_read_anchor` (read N lines centered on a named anchor without loading the full file). All reads use `readline` streaming. Anchor syntax: `@AI:NAV[(SEC|END|INS):id] description`. See `docs/ai-nav-anchors-plan.md`.
- **Session Analytics:** `src/SessionAnalyticsManager.ts` exposes `session_context_health` (context size + cost/turn + compact recommendation), `session_analyze_replay` (tool usage + retry signals across `~/.evokore/sessions/*-replay.jsonl`), and `session_work_ratio` (evidence-to-replay density scorer). See `docs/ai-nav-anchors-plan.md` Session Analytics section.
- **Skill Composition Graph:** `scripts/derive-skill-composition.js` emits `skill-graph.json`; `execute_skill` responses include `nextSteps[]`; `src/index.ts` auto-activates referenced tools and emits one `tools/list_changed`.
